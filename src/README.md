# Architecture

This project contains the following sub-modules / components:

1. Embedded MicroPython Interpreter
2. OOP based Networking Library (WiFi SoftAP)
3. Peripherals (Display and LED control)
4. Filesystem Control (Fast Static Access)
5. SSD1306 GPIO Base "Driver"

## 1. Embedded MicroPython

The `components/micropython` module provides the thin C++ integration layer around the patched embedded MicroPython runtime. It initializes the interpreter heap and stack, exposes a small `MicroPython` class, and forwards C++ calls into the embedded interpreter through `mp_embed_exec_string_function()`.

In the current application flow, the main program loads the bundled PONG MicroPython code at startup and then repeatedly calls a render function with JSON-style string parameters. This keeps the game logic inside MicroPython while the surrounding ESP-IDF application remains responsible for networking, task startup, and hardware control.

## 2. OOP Based Network Library

The networking code lives in `components/network_oop` and wraps the ESP-IDF / lwIP socket APIs in small C++ classes. It contains the HTTP server entry point, client and client-handler abstractions, and the Wi-Fi SoftAP setup used by the project.

At runtime the firmware starts a dedicated HTTP server task, configures a static SoftAP address, and tracks station connect or disconnect events through FreeRTOS event groups. Those events are then consumed by the rest of the application to update the title screen state and trigger LED feedback.

## 3. Peripherals

The `components/peripherals` module groups the user-interface related hardware abstractions. It provides display handling, title screen and game screen rendering, GPIO and PWM based LED helpers, and the cube animation used by the startup screen.

These classes are used by `main/micropython_as.cpp` to present system state directly on the SSD1306 display while also controlling the three-color status LED. As a result, player join or leave events, score flashes, and the transition from title screen to game screen are all handled by reusable C++ peripheral classes instead of inline application logic.

## 4. Filesystem Integration

The filesystem component implements a static in-flash file store for the HTTP server instead of relying on SPIFFS or LittleFS. A small metadata layer maps URL paths to pre-generated byte arrays, allowing the server to serve files such as `/index.html` directly from flash memory.

This design keeps the firmware lightweight and avoids runtime file I/O and buffer copies. The included conversion tooling generates `filedata.h` and `filemetadata.h` from the source files in `components/filesystem/filesystem`, so web assets become part of the normal firmware build.

## 5. SSD Base Driver

The low-level SSD1306 support is isolated in `components/ssd1306`. This is a reduced copy of the upstream ESP-IDF SSD1306 library, trimmed to the text, pixel, and line-drawing functions required by the project.

The higher-level `Display` class from the peripherals module wraps this C driver and fixes the board-specific I2C pin configuration for the currently supported ESP32-S3 setup. Keeping the raw driver separate from the C++ wrappers makes it easier to reuse the proven low-level implementation while keeping the application code clean.
